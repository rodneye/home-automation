Why Bother
02 / 12

The problem was not running containers. The problem was staying sane.

Docker and Docker Compose were fine right up until the stack started growing and the whole thing became a YAML hedge maze with too many places to forget something important.

I wanted home services that were reliable, scalable enough to grow, easier to manage, and useful as a way to learn more than "restart the container and hope for the best."

What needed to improve

  • Reliable service hosting for things I actually use at home.
  • A cleaner way to scale beyond one more Compose file.
  • Operations that do not depend entirely on memory and luck.

What I wanted out of it

  • Centralized ingress, storage, and deployment patterns.
  • A platform that is easier to grow than a pile of containers.
  • A good excuse to learn Kubernetes without renting a cloud identity crisis.
Compose was not bad. It just stopped being the shape of the problem.
The Layout
03 / 12

The actual shape of it

Users, DNS, ingress, workloads, and storage all have a clear place, which is already a major improvement over "I think that container was important."

Small hardware, proper architecture, fewer mystery cables in software form

Core layout

  • K3s control-plane and worker nodes run the workloads.
  • Traefik handles ingress inside the cluster.
  • NFS handles persistent storage outside the nodes.
  • Cloudflare Tunnel fronts external access cleanly.

Why this helps

  • Each concern has a job instead of every host doing everything badly.
  • The stack is easier to reason about when something breaks.
  • The architecture scales better than adding more Compose files and confidence.
Once the architecture is clear, the hardware mostly just has to avoid drama.

Why This Exists
05 / 12

The goal was not a mini cloud, just a sane home setup

This setup exists to run useful workloads at home without turning the rack, the office, or the maintenance burden into a second job.

Design constraints

  • Low-power Raspberry Pi and Debian hardware, not enterprise gear.
  • Limited appetite for manual babysitting and repeated shell archaeology.
  • Remote access needed to be clean without punching holes in the house.
  • Configuration had to be declarative enough to survive my own decisions.
Useful Home automation and cluster services that are worth keeping online.
Manageable Simple enough that fixing one bad evening does not create three more.

Environment anchors

  • Single control-plane node on the home network.
  • Dedicated NFS server for persistent storage.
  • Shared domain and Cloudflare-managed external access.
The README values are examples, not laws of physics.
Lesson One
06 / 12

Kubernetes is easy right up until the hosts are wrong

Install the base packages

Start with boring host prep because skipping boring host prep is how you invent new problems.

Disable swap

Turn off active swap and disable the services behind it, because Kubernetes will absolutely notice.

Enable Raspberry Pi memory cgroups

Append `cgroup_memory=1 cgroup_enable=memory` and avoid a deeply uninteresting debugging session later.

Base setup

  • Bring the node up to date.
  • Install the minimum packages for K3s and NFS support.
  • Keep the host layer intentionally boring.

Critical validation

  • Confirm swap is fully disabled.
  • Confirm memory cgroups are active on Raspberry Pi nodes.
  • Validate the host before blaming Kubernetes.
K3s is lightweight, not magical. Most early cluster weirdness came from the host layer being slightly wrong in ways that felt harmless until they were not.
This is the stage where you eliminate host-level drift.
Lesson Two
07 / 12

Shared storage first, cluster heroics second

NFS flow

  • Create `/mnt/nfs-storage` on every node that needs access.
  • Mount the export from `192.168.88.100:/export/k8s-storage`.
  • Validate the mount with `findmnt` and `df -h`.
  • Use the NFS subdir external provisioner for dynamic PVCs.

K3s install sequence

  • Install the control-plane node with the K3s bootstrap script.
  • Read the join token from `/var/lib/rancher/k3s/server/node-token`.
  • Join each worker with `K3S_URL` and `K3S_TOKEN`.

Control-plane install

  • Bootstrap a single control-plane node first.
  • Verify the cluster is healthy before adding workers.

Worker join

  • Join workers using the control-plane endpoint and token.
  • Treat workers as replaceable infrastructure.

Dynamic storage

  • Expose NFS into the cluster as a dynamic storage class.
  • Keep persistent state off the nodes themselves.
Persistent storage is where "home lab" stops being a toy and starts demanding consequences.
Lesson Two Point Five
08 / 12

Operational truth

DNS remains undefeated. It is either the problem or about to become the problem.

K3s was easy. DNS was the part that wanted opinions.

I went with Cloudflare Zero Trust because I did not want DNS living in one place, ingress rules in another, and my sanity filing for separation.

Why this route

  • The tunnel routes traffic straight into the cluster without opening inbound access.
  • DNS stays tied to the same Cloudflare setup that fronts the workloads.
  • I do not have to manage public DNS separately from what the cluster is actually serving.

Security shape

  • Access is protected with a Zero Trust application policy.
  • In this case the token flow is linked back to my email identity.
  • The result is private access with fewer homemade security mistakes.
If traffic is not arriving, it is probably DNS. If it is arriving, DNS is taking credit.
Lesson Three
09 / 12

Traefik inside, Cloudflare outside, fewer regrettable firewall ideas

The easiest remote-access win was deciding not to expose home services directly and letting the tunnel carry the awkward part of the job.

Zero Trust tunnel

Create and authenticate the tunnel, then let Kubernetes consume the credentials as a secret.

DNS routing

Bind the wildcard hostname to the tunnel so the cluster feels coherent instead of improvised.

Operational pattern

  • Create a Cloudflare Tunnel and authenticate it once.
  • Store tunnel credentials in Kubernetes as a secret.
  • Route the chosen domain through Cloudflare instead of opening inbound access.
Source IP preservation still needs `externalTrafficPolicy: Local` on Traefik.
Lesson Four
10 / 12

Argo CD in the cluster

The repo includes ingress and config so Argo CD can sit behind Traefik and stop deployment drift from becoming a hobby.

Argo CD is where the YAML therapy becomes at least somewhat supervised

Install the namespace and core components

Apply the upstream install manifest into the `argocd` namespace and get the control plane in place.

Apply the local server config

`argocd/configmap.yaml` lets `argocd-server` run behind Traefik without extra drama.

Expose the UI

Use `argocd/argocd-ingress.yaml`, grab the initial admin secret, and start managing apps from Git instead of memory.

Once Argo CD is live, cluster apps stop being ad hoc shell history and become declarative state.
What Broke
11 / 12

What needed checking, what kept going sideways

The cluster is small enough to understand quickly, but only if the checks are routine and the failure modes are treated like engineering rather than vibes.

The recurring lesson was simple: disposable workers are fine, hidden state is not. If a node matters, its data should already live somewhere else.

Cluster health

  • Know which nodes are healthy.
  • Know which workloads are actually running.

K3s services

  • Separate control-plane issues from worker issues quickly.
  • Keep service-level failures visible and obvious.

NFS verification

  • Verify that storage is mounted where you expect it.
  • Assume broken storage before assuming broken apps.

Troubleshooting trail

  • Follow service logs first.
  • Use host diagnostics when the problem smells lower-level.
This stack stays maintainable because the failure domains are small and visible.
Takeaways
12 / 12

Useful, manageable, and just dangerous enough to be fun

  • Prepare the hosts correctly before installing Kubernetes.
  • Use NFS-backed storage to keep persistent state off the nodes.
  • Expose services through Traefik and Cloudflare Tunnel instead of direct inbound access.
  • Use Argo CD to move from bootstrap commands into GitOps operations.

Talk landing point

This is not a perfect platform and that is part of the point. It is a practical stack built from real tradeoffs, mildly painful lessons, and tools that reduce unnecessary complexity.
